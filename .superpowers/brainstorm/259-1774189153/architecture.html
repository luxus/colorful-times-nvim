Three rewrite approaches

Same feature set, different structural philosophy. Pick what feels right.

A

Monolith-with-modules (recommended)

Single plugin, cleanly split into focused modules: core.lua (state machine + timers), schedule.lua (parsing + matching), system.lua (OS detection), tui.lua (floating window + picker), state.lua (JSON persistence). Everything lazy-loaded. No external deps.

Pros

  • Zero dependencies
  • Fast, self-contained
  • Easy to test each module
  • Simple for users to install

Cons

  • More code to own
  • Picker built from scratch
B

Lean core + snacks.nvim TUI

Keep the scheduling engine lean, but delegate the TUI (picker, input forms, notifications) to snacks.nvim (Folke's modern UI library). snacks provides a native fuzzy picker, input widgets, and notifications out of the box.

Pros

  • Less UI code to write
  • Polished UI for free
  • snacks is widely adopted

Cons

  • Hard dependency on snacks
  • Less control over UX
  • snacks API may change
C

vim.ui.select / vim.ui.input only

Zero custom TUI. All user interaction goes through vim.ui.select and vim.ui.input — which the user's own UI plugin (Telescope, fzf-lua, snacks) overrides automatically. Dead simple implementation.

Pros

  • Minimal code
  • Respects user's UI setup
  • Future-proof

Cons

  • No custom fuzzy preview
  • Multi-step flow is awkward
  • Less "TUI feel"
